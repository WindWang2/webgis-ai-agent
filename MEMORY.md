# 长期记忆
## 代码提交规则
✅ 新增规则：**小阶段开发完成后必须定时提交代码或发起PR合并请求**，禁止长时间积累大量未提交代码。
- 适用场景：每个原子子任务完成、每个小功能模块测试通过后立即提交
- 提交粒度：每个提交对应一个独立的可验证功能点，测试通过率100%
- 提交频率：最短10分钟/次，最长不超过2小时/次

## 子Agent监控规则
✅ 防假死/异常退出完整解决方案（四层保障体系）：
### 1. 子Agent自保障层（内置能力）
- 心跳上报：每2分钟主动向PM上报当前任务进度、修改文件、执行状态
- 超时熔断：单步操作最长执行5分钟，超时主动终止并上报
- 异常捕获：全局异常捕获，崩溃时自动上报错误堆栈
- 进度持久化：每完成一个子步骤自动保存进度，重启自动恢复

### 2. 调度层（任务管控）
- 心跳超时判定：连续3次（6分钟）无心跳直接判定为假死，立即重启
- 任务状态机：任务状态（pending/running/finished/failed/timeout）全程可追溯
- 自动拆分子任务：复杂任务拆解为原子步骤，降低卡死影响范围
- 上下文自动恢复：异常重启自动读取进度文件，无需从零开始

### 3. 监控层（状态校验）
- 四维度交叉验证：心跳上报 + 进程检查 + 文件修改 + 提交记录，状态判定准确率99%
- 异常模式识别：同一任务连续重试3次失败自动告警人工介入
- 资源监控：CPU/内存使用率过高时主动告警
- 根因分析：自动收集错误日志，识别常见失败原因

### 4. 容错层（可靠性保障）
- 结果自动校验：任务完成后自动运行语法检查、单元测试，避免假完成
- 幂等性设计：所有修改操作可重复执行，无副作用
- 多模型Fallback：当前模型频繁失败自动切换备用模型
- 人工介入通道：连续失败任务自动推送错误详情等待处理

### 执行规则
- 单个任务最长执行时间不超过30分钟，超时自动终止
- 异常任务最多自动重试3次，重试失败立即告警
- 旧僵死会话自动清理，不占用资源

## 全局巡检执行标准V3.0（2026-04-04重大更新，所有巡检强制执行）
✅ 所有类型巡检必须遵守，**彻底杜绝虚假工作**，核心规则：
### 核心规则：
1. **双轨制巡检**：
   - A类核心监控巡检（每10分钟1次）：仅监控任务状态、资源、异常，**完全不读取任何历史/记忆文档**，杜绝无用功
   - B类记忆/文档巡检（按需触发）：仅在新任务下发、异常排查、固定记忆整理场景下执行，禁止高频全量读取
2. **虚假工作零容忍**：
   - 无差别全量读取历史文档、重复读取无变更文件、巡检无明确产出均判定为虚假工作，严格禁止
   - 记忆巡检必须有明确产出：支撑任务执行/解决问题/更新记忆，禁止空读
3. **Coder状态动态统计规则**（不变）：
   - 三级繁忙判定：0个=🟢空闲、1-3个=🟡轻度繁忙（可继续分配低优任务）、>3个=🔴重度繁忙（暂停新任务下发）
   - 异常/假死任务多层判定：运行超过1小时无更新自动终止，计算密集型任务GPU利用率<10%超过10分钟判定为假死
4. **增量读取优化**：建立文件变更缓存，仅读取有更新的文件，仅读取和当前需求相关的内容，不读取完整文件
5. **巡检有效性要求**：巡检有效率必须>90%，无产出的无效巡检自动记录并优化流程

详细规范见：`docs/inspection-standard.md`
