## 1. 项目概述

### 1.1 项目背景与目标

随着城市化进程加速和空间数据的爆发式增长，GIS（地理信息系统）分析的需求日益增加。然而，传统 GIS 工具对用户的专业技能要求较高，非 GIS 专业人员难以进行复杂的空间数据分析。大语言模型（LLM）的突破性进展为这一问题提供了新的解决路径——通过自然语言交互驱动空间分析全链路，大幅降低 GIS 分析的使用门槛。

本项目旨在构建一个基于 Web 的 AI 智能体应用，用户通过自然语言指令即可完成从数据获取、空间分析到专题制图和报告生成的完整工作流。

**典型应用场景**：用户输入"分析最近降雨对武汉市道路交通的影响"，系统自动完成降雨数据获取、路网叠加分析、影响评估、主题地图生成以及多模态分析报告输出。

### 1.2 目标用户

- **主要用户**：城市规划师、应急管理人员、政府决策者等需要 GIS 分析能力但缺乏深度 GIS 操作经验的专业人员
- **次要用户**：GIS 从业者——用自然语言加速日常分析流程，提升工作效率
- **潜在用户**：科研人员、新闻记者、企业分析师等需要空间分析辅助决策的群体

### 1.3 核心功能概览

| 功能模块 | 核心能力 | 输出物 |
|---------|---------|--------|
| 自然语言理解 | 任务拆解、意图识别、多轮对话 | 结构化执行计划 |
| 数据获取 | 多源异构地理数据自动采集 | 标准化图层数据 |
| 空间分析 | 缓冲区、叠加、插值、网络分析等 | 分析结果图层与统计数据 |
| 智能制图 | 自动专题图生成与样式优化 | 交互地图 + 静态图片 |
| 报告生成 | 图文混排、多模态分析报告 | HTML / PDF / Word 报告 |

## 2. 整体技术架构

### 2.1 架构总览

系统采用前后端分离的分层架构，整体分为四个核心层级：

- **前端展示层**：基于 React/Next.js 的 Web 应用，集成对话面板、地图面板和报告面板，提供流畅的用户交互体验。
- **API 网关层**：基于 FastAPI 构建的 RESTful + WebSocket 服务，负责请求路由、认证鉴权、流式通信。
- **Agent 编排层**：核心智能体引擎，基于 LLM（Claude API）的 Tool Use 机制，负责任务拆解、工具调度和结果汇总。
- **工具服务层**：包含数据获取、空间分析、制图、报告生成四大类工具集，每个工具封装为独立的可调用单元。

### 2.2 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|-----|---------|---------|
| 前端框架 | Next.js 14 + React 18 | SSR 支持、App Router、生态成熟 |
| 地图引擎 | MapLibre GL JS | 开源免费、高性能 WebGL 渲染、自定义底图 |
| UI 组件库 | Ant Design / shadcn/ui | 企业级组件生态、可定制性强 |
| 后端框架 | FastAPI (Python) | 异步支持、与 GIS 生态无缝衔接、自动 API 文档 |
| LLM 引擎 | Claude API (Tool Use) | 强大的工具调用能力、长上下文窗口、推理精准 |
| 空间分析 | GeoPandas + Shapely + Rasterio | Python GIS 生态核心、功能全面 |
| 空间数据库 | PostGIS (PostgreSQL) | 专业空间索引、复杂空间查询 |
| 消息队列 | Celery + Redis | 异步任务处理、长时间分析任务支持 |
| 对象存储 | MinIO / AWS S3 | 分析结果、地图图片、报告文件存储 |
| 容器化 | Docker + Docker Compose | 环境一致性、一键部署 |

## 3. MCP（Model Context Protocol）方案

### 3.1 MCP 概述与必要性分析

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，定义了 LLM 与外部工具/数据源之间的标准化通信接口。在本项目中，MCP 的引入不是必需的，但在特定场景下能带来显著的架构优势。

### 3.2 是否需要 MCP：分场景评估

| 场景 | 不用 MCP | 用 MCP | 建议 |
|-----|---------|-------|------|
| 调用天气/交通 API | 直接定义 Tool 即可 | 封装为 MCP Server | MVP：不需要 |
| 调用空间分析工具 | Python 函数直接调用 | 封装为 MCP Tool | MVP：不需要 |
| 多 Agent 协作/工具共享 | 各 Agent 各自维护 Tool | 统一 MCP Server 复用 | 扩展阶段引入 |
| 第三方工具集成 | 逐个写 Adapter | 接入社区 MCP Server | 有现成 Server 时用 |
| 插件化扩展 | 需定义插件协议 | 接入现有 MCP 生态 | 扩展阶段引入 |
| 用户部署 MCP Server | 平台化阶段 | 允许用户注册自定义工具 | 平台化阶段 |

### 3.3 MCP 引入策略

**阶段一（MVP）**：不引入 MCP。直接使用 Claude API 的原生 Tool Use 机制，将数据获取和分析工具定义为 Python 函数，通过 Function Calling 调用。这是最简单直接的方案，开发成本最低。

**阶段二（扩展期）**：选择性引入 MCP。当工具数量超过 20 个、需要跨服务共享工具定义时，将核心工具层重构为 MCP Server。此时 Agent 通过 MCP 协议与工具通信，架构解耦更彻底。

**阶段三（平台化）**：全面 MCP 化。对外暴露 MCP 接口，允许用户和第三方开发者注册自定义数据源和分析工具，形成开放的 GIS 工具生态。

### 3.4 MCP 架构设计（扩展阶段）

当决定引入 MCP 时，建议按功能域拆分为多个 MCP Server：

| MCP Server | 职责 | 暴露的 Tools | 示例 |
|-----------|------|-------------|------|
| gis-data-server | 封装所有外部数据获取能力 | fetch_weather, fetch_traffic, fetch_osm, fetch_dem | 天气、交通、OSM、DEM |
| gis-analysis-server | 封装空间分析能力 | buffer, overlay, interpolation, zonal_stats | 缓冲区、叠加、插值、分区统计 |
| gis-cartography-server | 封装制图能力 | create_thematic_map, export_map_image | 专题图、地图导出 |
| gis-report-server | 封装报告生成能力 | generate_report, export_pdf, export_docx | 报告生成、PDF/Word 导出 |
| gis-db-server | PostGIS 数据库查询能力 | query_spatial, insert_layer, list_tables | 空间查询、图层管理 |

## 4. RAG（检索增强生成）方案

### 4.1 RAG 概述与必要性分析

RAG（Retrieval-Augmented Generation）通过在 LLM 生成前检索相关知识片段注入上下文，显著提升回答的准确性和领域专业性。在本项目中，RAG 的引入是高度推荐的，原因如下：

- **GIS 分析方法论知识**：LLM 虽然具备通用知识，但对具体的 GIS 分析流程缺乏系统性的专业深度，需要通过 RAG 补充领域知识。
- **API 文档与参数规范**：各数据源 API 的接口文档、参数说明、返回格式等信息经常变化，RAG 可以确保 Agent 始终使用最新的 API 调用方式。
- **分析模板与最佳实践**：预置的分析 workflow 模板可以通过 RAG 动态检索，指导 Agent 的任务规划。
- **用户历史分析复用**：用户过去做过的分析报告和参数配置可以被检索复用，实现个性化的分析建议。

### 4.2 RAG 知识库设计

知识库按用途和更新频率分为四层：

#### 4.2.1 GIS 方法论知识库（静态，专家维护）

| 知识类型 | 内容示例 | 来源 |
|---------|---------|------|
| 分析方法论 | 城市内涝分析流程、交通影响评估方法 | GIS 教材、技术规范 |
| 空间分析算法 | 各种算法的适用场景、参数含义、结果解读 | 算法文档、学术论文 |
| 制图规范 | 专题图配色规范、分级方法选择指南 | 制图学教材、OGC 规范 |
| 指标体系 | 交通拥堵指数、洪涝风险等级等标准定义 | 国标、行业标准 |

#### 4.2.2 API 文档知识库（半动态，定期更新）

| 知识类型 | 内容示例 | 更新频率 |
|---------|---------|---------|
| 数据源 API 文档 | 和风天气 API 参数、高德交通 API 返回格式 | 月度 |
| Tool 参数规范 | 每个 Tool 的输入输出 Schema、调用约束 | 随代码更新 |
| 数据字典 | 各数据源的字段含义、单位、坐标系 | 季度 |

#### 4.2.3 分析模板库（动态，持续积累）

预置一系列典型分析场景的 workflow 模板，每个模板包含标准的分析步骤、所需数据、推荐工具组合和报告框架：

- 降雨 - 交通影响分析模板
- 城市内涝风险评估模板
- 人口热力分析模板
- 环境污染扩散分析模板
- 城市规划选址评估模板
- 灾害应急路径规划模板

#### 4.2.4 用户历史知识库（动态，自动生成）

- 历史分析记录：用户过去执行过的分析任务及其参数配置
- 自定义数据源配置：用户注册的私有数据接口信息
- 个性化偏好：配色偏好、默认坐标系、常用分析区域等

### 4.3 RAG 技术实现

| 组件 | 技术选型 | 说明 |
|-----|---------|------|
| 文档解析 | LangChain Document Loaders | 支持 PDF、Markdown、HTML 等多种格式 |
| 文本分块 | RecursiveCharacterTextSplitter | 按语义边界分块，chunk_size=1000 |
| 向量嵌入 | text-embedding-3-small / BGE-M3 | 支持中英双语，BGE-M3 可本地部署 |
| 向量数据库 | pgvector (PostgreSQL 扩展) | 与 PostGIS 共用 PostgreSQL 实例 |
| 检索策略 | Hybrid Search (向量 + BM25) | 兼顾语义相似和关键词精确匹配 |
| 重排序 | Cohere Reranker / BGE-Reranker | 对初检结果精排，提升 Top-K 相关性 |

### 4.4 RAG 引入策略

**阶段一（MVP）**：轻量 RAG + 提示词缓存（Prompt Caching）。将 GIS 方法论知识和 API 文档以结构化 Markdown 格式编写，直接作为 System Prompt 的一部分注入 LLM 上下文。为降低计算延迟和 Token 成本，**必须应用大模型原生的 Prompt Caching 特性**，将大规模静态 API 规范标记为 `ephemeral`，以便多轮对话极速复用计算层。不依赖庞大的外部向量系统。

**阶段二（正式版）**：完整 RAG 管线。部署向量数据库，建立完整的知识检索管线。用户查询时，先检索相关知识片段，再与用户指令一起发送给 LLM。注意隔离用户级数据与系统级专家数据。

**阶段三（平台化）**：用户级知识库。每个用户拥有独立的知识空间，历史分析自动入库，支持团队共享知识库。引入知识质量评估和过期清理机制。

## 5. Agent 编排层设计

### 5.1 Agent 工作流

Agent 的核心工作流遵循 Plan-Act-Observe 循环：

1. **接收指令**：用户输入自然语言指令，Agent 接收并解析意图。
2. **知识检索（RAG）**：根据识别到的分析主题，从知识库检索相关方法论和分析模板。
3. **任务规划**：LLM 基于用户指令和检索到的知识，生成结构化的执行计划。
4. **工具调用**：按计划依次调用数据获取、空间分析、制图等工具，每步收集中间结果。
5. **结果校验与强验证（Self-Correction）**：所有 Tool 执行前，使用严密的 Pydantic 模型作为护栏验证入参。若因大模型产生“幻觉”（如参数缺失或坐标非法）导致无法执行计算，引擎**原封不动抛回 Exception 信息**（充当思考链），指示大模型自动重试修复，实现程序的“自愈”而不是抛错到前端。
6. **迭代优化**：如果结果不满意或工具调用反复失败，Agent 自动调整宏观分析策略并重试。
7. **结果汇总**：整合所有分析结果，生成最终的文字结论、主题地图和分析报告。

### 5.2 Tool Schema 设计原则

- **name**：工具唯一标识符，如 `fetch_weather`、`buffer_analysis`
- **description**：清晰描述工具的功能、适用场景和使用约束
- **input_schema**：应用 Pydantic 生成基于 JSON Schema 的强制结构，**杜绝弱类型绑定**
- **output_schema**：返回结果的结构定义，确保 Agent 能正确解析并能映射为标准字典
- **数据黑盒传递原则**：绝不允许向大模型上下文中通过 JSON 原文倾倒 `10k+` 的坐标长文本序列。任何大型的 `GeoJSON/Shapefile` 仅使用游标句柄（如 `layer_id: "tmp_abc123"`）在大模型交互指令中代付传递。
- **error_codes**：可能的报错类型与建议补救方法，帮助模型自愈

### 5.3 System Prompt 设计要点

System Prompt 是 Agent 行为的核心准则，需要精心设计以下几个方面：

- **角色定义**：明确 Agent 是一个 GIS 分析助手，擅长空间数据分析和制图
- **能力边界**：明确告知 LLM 可用的工具列表和每个工具的能力范围
- **分析范式**：注入常见的 GIS 分析模式（如影响分析 = 叠加分析 + 缓冲区 + 统计）
- **质量约束**：要求所有数值结论必须来自工具计算结果，禁止 LLM 编造数据
- **交互规范**：遇到歧义时应主动询问用户，而不是自行假设

## 6. 数据获取层设计

### 6.1 数据源清单与接入方案

| 数据类别 | 数据源 | 接口方式 | 数据格式 |
|---------|-------|---------|---------|
| 气象数据 | 和风天气 / OpenWeatherMap / ERA5 | REST API / CDS API | JSON → GeoJSON |
| 交通数据 | 高德交通 / 百度路况 / TomTom | REST API | JSON → GeoJSON |
| 基础底图 | OpenStreetMap / Overpass API | REST API | OSM XML → GeoJSON |
| 行政边界 | 天地图 / Natural Earth / GADM | REST API / 静态文件 | GeoJSON / Shapefile |
| 高程数据 | OpenTopography / SRTM | REST API | GeoTIFF |
| 卫星影像 | Sentinel Hub / Google Earth Engine | REST API / SDK | GeoTIFF / PNG |
| 社会经济 | 国家统计局 / WorldPop | REST API / 下载 | CSV / GeoTIFF |
| 用户上传 | 本地文件上传 | multipart/form-data | Shapefile/GeoJSON/CSV |

### 6.2 统一数据接入层（DataSource Registry）

所有数据源通过统一的注册表管理，每个数据源实现标准接口。注册表维护数据源的元信息（名称、描述、参数 Schema、API Key 配置、速率限制等），Agent 通过查询注册表获得可用数据源列表和调用方式。

数据源返回的结果统一转换为标准化的 GeoJSON 或 GeoTIFF 格式，附带 CRS、时间戳和质量元数据。

## 7. 空间分析引擎

### 7.1 分析能力矩阵

| 分析类型 | 实现方案 | 典型应用场景 |
|---------|---------|-------------|
| 缓冲区分析 | Shapely buffer / PostGIS ST_Buffer | 河流影响范围、道路服务区 |
| 叠加分析 | GeoPandas overlay / PostGIS ST_Intersection | 降雨区域与路网交叉分析 |
| 空间插值 | SciPy IDW / Kriging / GDAL Grid | 气象站点数据→连续面 |
| 网络分析 | NetworkX / OSRM / pgRouting | 最短路径、应急路线规划 |
| 热力图/密度分析 | KDE (SciPy) / H3 六边形聚合 | 事故聚集区、人口密度统计 |
| 汇总分析 | GeoPandas + Pandas | 分区统计、分级频率分析 |
| 栅格计算 | Rasterio + NumPy | NDVI 计算、坡度分析、变化检测 |
| 地址编码 | 高德/百度地理编码 API | 地址→坐标转换 |
| 坐标变换 | pyproj | WGS84/CGCS2000/Web Mercator 互转 |

### 7.2 分析工具封装原则

- **原子性**：每个 Tool 只做一件事，复杂分析由 Agent 编排多个 Tool 组合完成
- **计算隔离原则（非阻塞）**：GIS 计算为重度 CPU 密集型操作。凡是涉及大型地理文件解析或节点超十万级别的 GeoPandas/Rasterio 计算，**MVP 阶段也严禁**直接暴露在 FastAPI 异步路由阻塞主进程，必须由挂载在后端的 Celery Worker 和 Redis 消息队列进行分布式处理。
- **确定性**：相同输入保证相同输出，所有数值计算由代码而非 LLM 完成
- **可追溯**：每个 Tool 记录输入参数、执行时间、输出结果摘要，保证状态追踪
- **可降级**：当首选实现不可用时，自动降级到备选方案

### 8.2 静态出图方案

除了 Web 交互式地图外，需要生成适合嵌入报告的静态地图图片：

- **方案 A（推荐）**：后端使用 Matplotlib + Cartopy / contextily 生成出版级静态地图
- **方案 B**：前端 MapLibre 渲染后，通过 Puppeteer/Playwright 无头浏览器截图
- **配色方案**：集成 ColorBrewer 色板，Agent 根据数据类型自动选择合适的色系

## 9. 报告生成引擎

### 9.1 报告内容结构

标准分析报告的结构模板如下：

1. **概述**（分析目标、时空范围、数据来源说明）
2. **数据概况**（使用的数据集、数据质量评估、预处理说明）
3. **分析方法**（采用的分析方法及参数说明）
4. **分析结果**（关键发现，配以主题地图和统计图表）
5. **结论与建议**（基于分析结果的结论和行动建议）
6. **附录**（数据详表、方法引用、数据源链接）

### 9.2 报告技术方案

| 组件 | 技术方案 | 说明 |
|-----|---------|------|
| 文字内容 | LLM 生成 | 基于分析结果的结构化 JSON 撰写专业文字 |
| 统计图表 | ECharts / Plotly | 柱状图、折线图、饼图等数据可视化 |
| 主题地图 | Matplotlib 静态图 / MapLibre 截图 | 嵌入报告的地图图片 |
| 模板渲染 | Jinja2 HTML 模板 | 统一的报告版式和样式 |
| PDF 导出 | WeasyPrint | HTML→PDF，支持分页、页眉页脚 |
| Word 导出 | python-docx | 结构化的 Word 文档输出 |
| 在线预览 | React HTML 渲染 | 前端直接预览 HTML 版报告 |

## 10. 前端交互设计

### 10.1 界面布局

前端采用三栏式布局：

- **左侧面板**：AI 对话区域，包含消息流、输入框、文件上传区。支持 Markdown 渲染、图片内嵌。展示 Agent 的思考链和工具调用进度。
- **中央区域**：地图展示区，基于 MapLibre GL JS 的交互式地图。支持图层管理、属性查询、空间量测工具。
- **右侧面板（可收起）**：报告预览区/图层属性表/分析参数面板。根据当前操作阶段动态切换内容。

### 10.2 实时通信方案

前后端通过 WebSocket 保持长连接，实现以下实时能力：

- LLM 回复的流式输出（打字机效果）
- Agent 工具调用进度的实时推送
- 地图图层的动态加载通知
- 长时间分析任务的进度条更新

## 11. 部署与基础设施

### 11.1 部署架构

| 组件 | 部署方式 | 资源需求 |
|-----|---------|---------|
| 前端 (Next.js) | Vercel / Nginx 静态托管 | 低（CDN 分发） |
| 后端 (FastAPI) | Docker 容器 + Gunicorn | 2 核 4G 起步 |
| PostgreSQL + PostGIS + pgvector | Docker 容器 / 云数据库 | 4G 内存起步 |
| Redis | Docker 容器 / 云 Redis | 1G 内存 |
| Celery Worker | Docker 容器，可水平扩展 | 按任务量扩展 |
| MinIO / S3 | 自建 MinIO / 云对象存储 | 按存储量计费 |

### 11.2 Docker Compose 服务编排

使用 Docker Compose 统一编排所有服务，确保开发环境、测试环境和生产环境的一致性。核心服务包括：

- `web`（FastAPI 主服务）
- `worker`（Celery 异步任务）
- `db`（PostgreSQL + PostGIS + pgvector）
- `redis`（消息队列和缓存）
- `minio`（文件存储）

开发环境增加 pgAdmin 和 Flower 监控。

## 12. 开发路线图

### 12.1 Phase 1：MVP（第 1-6 周）

**目标**：单一场景全链路打通

- 搭建前后端基础框架（FastAPI + Next.js + MapLibre）
- 实现 Agent 核心循环（Claude Tool Use）
- 接入 2-3 个核心数据源（天气 API、OSM 路网、行政边界）
- 实现 3-4 个基础分析 Tool（叠加、缓冲区、统计汇总）
- 基础专题图生成（一种分级色彩图）
- 简单文字结论输出
- 轻量 RAG：将方法论知识直接写入 System Prompt

### 12.2 Phase 2：核心功能完善（第 7-14 周）

**目标**：多场景支持、完整制图和报告能力

- 扩展数据源至 5-8 个
- 扩展分析工具至 10+ 个
- 完善制图引擎，支持多种专题图类型
- 完整的报告生成管线（HTML + PDF + Word）
- 部署完整 RAG 管线（pgvector + 知识库）
- 用户上传数据支持
- 多轮对话上下文管理

### 12.3 Phase 3：平台化与优化（第 15 周+）

**目标**：产品级质量、可扩展的平台能力

- 引入 MCP 架构，工具层解耦为独立服务
- 用户系统、权限管理、使用计量
- 历史分析管理和复用
- 性能优化（缓存、异步任务、大数据量处理）
- 开放 API，支持第三方集成
- 团队协作和知识共享功能

## 13. 关键风险与应对策略

| 风险 | 影响 | 应对策略 |
|-----|------|---------|
| LLM 任务拆解不准确 | 分析结果错误或流程中断 | Guardrails 校验层；人工介入修正；Few-shot 样例优化 |
| 数据源 API 不稳定 | 数据获取失败 | 备选数据源；统一错误处理和重试；API 健康监控 |
| LLM 编造分析结论 | 报告数据不可信 | 严格分离 LLM 和计算：数值必须来自代码结果 |
| 大数据量性能瓶颈 | 响应超时 | PostGIS 空间索引；数据分块处理；异步任务队列 |
| 制图质量不专业 | 影响产品可信度 | 预置专业制图模板和配色方案；引入制图规范知识库 |
| 坐标系不一致 | 空间分析结果偏差 | 统一数据中间层自动 CRS 转换；Tool 执行前校验 |

## 14. 总结

本技术方案描述了一个基于大语言模型的 WebGIS 智能分析系统的完整设计。系统的核心价值在于通过自然语言交互降低 GIS 分析的使用门槛，使非 GIS 专业人员也能完成专业的空间数据分析和制图工作。

在技术实现上，我们采用渐进式策略：MVP 阶段以最小成本验证核心价值——直接使用 Claude Tool Use 进行工具调度，将知识库以 Prompt 形式注入，快速打通全链路闭环。在验证可行性后，逐步引入完整的 RAG 管线和 MCP 架构，提升系统的专业性、可扩展性和开放性。

MCP 和 RAG 的引入都遵循"按需引入、渐进演化"的原则。MCP 适合在工具层复杂度提高后引入以实现架构解耦，RAG 则适合在知识库规模增长后引入以保持 Agent 的领域专业性。两者都不是 MVP 必需的，但都是系统走向成熟的重要技术储备。
